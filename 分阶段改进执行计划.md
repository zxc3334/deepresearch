# GeoResearch Agent 分阶段改进执行计划

本文档用于后续实际改造时跟踪进度。规则是：每个 phase 完成代码后，必须先经过 `code-review-expert` 检查，再跑单元测试/烟测，最后交给用户验收。用户验收通过后，才勾选完成并进入下一 phase。

## 总执行规则

- [ ] 每个 phase 开始前先确认改动范围，不混入无关功能。
- [ ] 每个 phase 完成后运行 `code-review-expert`。
- [ ] 每个 phase 完成后补充或更新单元测试。
- [ ] 每个 phase 完成后跑最小端到端 smoke test。
- [ ] 每个 phase 完成后更新 `项目日志.md`。
- [ ] 每个 phase 完成后由用户验收。
- [ ] 用户验收通过后再打勾，并进入下一 phase。

建议每个 phase 的提交节奏：

```powershell
git status -sb
git add <本阶段相关文件>
git commit -m "<阶段性提交信息>"
git push
```

`outputs/` 运行产物默认不提交，除非明确需要作为 demo artifact。

## Phase 3：Provider / Profile / ModuleProfile 模型架构优化

### 设计目标

```text
Provider: API 服务商、base_url、api_key、adapter
Profile: model、temperature、top_p、max_tokens、context limits
```

Policy 只负责一次 LLM 调用，尽量无状态。AgentPool 不复用带运行态污染的 policy。

## Phase 5.5：Tool Result Compact / Context Budget Manager

优先级：P1

目标：控制进入 LLM 的工具结果上下文，减少注意力稀释和 context rot，同时保留证据可追溯性。这个阶段关注“适配工具结果上下文”，不是直接复用 Orchestrator memory compact。

### 设计原则

```text
Tool execution
  -> 原始结构化 tool result
  -> ToolResultNormalizer
  -> ToolResultStore / trajectory / trace
  -> PromptBuilder
       -> ContextBudgetManager
       -> ToolResultCompactPolicy
  -> LLM messages
```

- 工具执行层尽量保留原始结构化结果。
- trace / trajectory 层保留可回溯信息。
- PromptBuilder / ContextBudgetManager 决定本次进入 LLM 的工具上下文预算。
- 被压缩内容必须标记 `[compact]`。
- error、failed、rejected、unsupported、约束类信息不压缩，避免重复犯错。

### 改动范围

- [x] 新增 `ToolResultNormalizer`，统一工具返回结构。
- [x] 新增 `ContextBudgetManager`，按 prompt section 管理预算。
- [x] 新增 `ToolResultCompactPolicy`：
  - [x] 保留 `title/url/source_tier/quality_score/evidence_level` 等结构化字段。
  - [x] 普通正文先做 head/tail 静态截断，默认 70/30。
  - [x] 超预算后按 `source_tier/quality_score/task relevance` 动态裁剪。
  - [x] 仍超预算时才允许 LLM 摘要作为后续可选扩展，本阶段默认不用 LLM 摘要。
  - [x] `general/speculative` 来源优先压缩或丢弃。
  - [x] `official/academic/evidence_backed` 优先保留。
- [x] PromptBuilder 接入 compact 后的 tool context。
- [x] trace 记录 compact 事件和 `[compact]` 标记。
- [x] YAML 暴露 context budget 参数：
  - [x] `tool_result_budget_chars`
  - [x] `synthesis_context_budget_tokens`
  - [x] `compact_trigger_ratio`
  - [x] `head_tail_ratio`

### 单元测试

- [x] 未超预算时不 compact。
- [x] 超预算时正文被 head/tail 截断并标记 `[compact]`。
- [x] error/failure/rejected 信息不会被压缩。
- [x] official/academic 结果优先保留。
- [x] general/speculative 结果优先被裁剪。
- [x] trace 中能看到 compact 事件。

### 验收标准

- [ ] 长工具结果不会显著稀释 prompt。
- [ ] 错误和约束信息不会因 compact 丢失。
- [ ] trace/report 能看出哪些内容被 compact。
- [ ] 用户确认通过。

## Phase 7：用户/会话隔离的长期记忆

优先级：P2

目标：理清 user/session/run/task 的关系，避免长期记忆互相污染。

### 数据关系

```text
user_id
  -> session_id
      -> run_id
          -> task_id
```

### 改动范围

- [ ] `SharedMemoryStore` 支持 `session_id`。
- [ ] `MemoryEntry` metadata 增加：
  - [ ] user_id
  - [ ] session_id
  - [ ] run_id
  - [ ] task_id
  - [ ] evidence_level
  - [ ] source_tier
- [ ] `query_by_similarity()` 支持 session/user 过滤。
- [ ] `get_context_for_query()` 支持 session/user 过滤。
- [ ] runner 支持传入 session_id。
- [ ] 失败结果精简写入 memory，避免重复犯错。

### SharedMemoryStore 存储边界

SharedMemoryStore 存：

- 成功任务的代表性 claim。
- verified/evidence_backed 结论摘要。
- rejected 结论和约束。
- failure memory。
- 用户偏好和会话指令摘要。

SharedMemoryStore 不存：

- 完整长报告。
- 大段网页原文。
- 未压缩工具结果。
- 未经过 gate 的 speculative 长文本。

- [ ] global memory 可以被所有 session 读取。
- [ ] failure memory 能按 query 检索。
- [ ] metadata 写入完整。

### 验收标准

- [ ] 多 session 运行不会互相污染。
- [ ] 用户确认通过。

## Phase 7.5：Model Presets / Module Model Selection

优先级：P2

目标：用户不填写 API key，只能从系统提供的模型方案中选择。每个方案定义 planner / researcher / summarizer / compressor 等模块的默认 profile，API key 由服务端环境变量统一管理。

### 设计原则

```text
Frontend selects model_preset
        |
        v
model.presets[preset].module_profiles
        |
        v
model.module_profiles override
        |
        v
LLMModelFactory
        |
        v
OpenAICompatiblePolicy
```

- 用户只看到 `balanced / low_cost / high_quality` 等模型方案。
- 用户不看到、不填写、不保存 API key。
- 服务端通过 `.env.local` 或部署环境变量保存 provider key。
- 每个模块有默认 profile，避免用户把不适合的模型分配给 planner/researcher/summarizer。
- 后续多用户场景只保存用户选择的 `model_preset`，不保存密钥。

### 改动范围

- [ ] YAML 新增 `model.presets` 和 `model.active_preset`。
- [ ] `LLMModelFactory` 支持将 preset 的 `module_profiles` 覆盖到运行时 module routing。
- [ ] 增加 `ProviderRegistry`，内置常见 provider 模板：
  - [ ] deepseek
  - [ ] openai
  - [ ] openrouter
  - [ ] siliconflow
  - [ ] kimi / moonshot
  - [ ] qwen / dashscope
- [ ] 增加 preset validation：
  - [ ] preset 必须存在。
  - [ ] preset.module_profiles 指向的 profile 必须存在。
  - [ ] profile 指向的 provider 必须存在。
- [ ] 前端/接口层只传 `model_preset`。
- [ ] trace/run metadata 记录 active preset 和模块 profile 映射，但不记录 API key。

### 单元测试

- [ ] `active_preset=balanced` 能生成正确 module_profiles。
- [ ] 用户选择 `high_quality` 能覆盖默认 preset。
- [ ] preset 引用不存在 profile 时错误明确。
- [ ] trace metadata 不包含 API key。
- [ ] 旧配置没有 presets 时仍按 `module_profiles` 工作。

### 验收标准

- [ ] 用户能在几个系统提供的模型方案中选择。
- [ ] 不需要也不能从前端填写 API key。

- [ ] 用户确认通过。

## Phase 8：最小 Wiki 知识库

优先级：P2

目标：把高质量研究结果沉淀为可读、可维护的 Markdown 知识库。

### Wiki 存储边界

Wiki 存：

- 领域实体知识，例如 sensor/method/dataset/analysis/project。
- 高质量研究报告 raw。
- 可复用结论。
- 被明确否定的约束知识，例如 Sentinel-2 不能直接反演 LST。

Wiki 不存：

- 每个 task 的完整 trajectory。
- 大段低质量网页摘录。
- provider/API 错误日志。
- 未达 evidence gate 的 speculative 结论。

### 改动范围

- [ ] 迁移最小 `WikiStore`。
- [ ] 支持用户目录：

```text
data/wiki/users/<user_id>/
  index.md
  raws/
  sensors/
  methods/
  analyses/
  comparisons/
  projects/
  notes/
```

- [ ] 支持 `save_raw()`。
- [ ] 支持 `get_index()`。
- [ ] 支持 `get_context(query)`。
- [ ] Orchestrator 在 planning/task context 中注入 wiki context。
- [ ] Wiki ingest 先只生成 draft，不自动 verified。

### Ingest gate

- [ ] report 不是失败 fallback。
- [ ] 至少一个成功子任务。
- [ ] confidence 达到阈值。
- [ ] 至少一个 `evidence_backed` 或 `verified` evidence。

### 单元测试

- [ ] 用户能人工检查 Wiki Markdown 文件。

## 当前推荐启动顺序

严格顺序：

1. Phase 1：证据质量统一升级
2. Phase 2：搜索结果质量排序
3. Phase 3：Provider / Profile / ModuleProfile 模型架构优化
4. Phase 4：可配置外部证据预取
5. Phase 5：Pipe-based PromptBuilder 增量重构
6. Phase 5.5：Tool Result Compact / Context Budget Manager
7. Phase 6：实时进度与用户调整方向
8. Phase 7：用户/会话隔离的长期记忆
9. Phase 7.5：Model Presets / Module Model Selection
10. Phase 8：最小 Wiki 知识库
11. Phase 9：Wiki ingest 与 lint 增强

排序理由：

- 先做 evidence/search，因为它们直接决定研究结果是否可信。
- 再做 model 架构，因为它影响后续所有 agent 的成本和质量控制。
- 再做 external prefetch，因为它依赖 evidence/search 的质量判断。
