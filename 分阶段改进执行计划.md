# GeoResearch Agent 分阶段改进执行计划

本文档用于后续实际改造时跟踪进度。规则是：每个 phase 完成代码后，必须先经过 `code-review-expert` 检查，再跑单元测试/烟测，最后交给用户验收。用户验收通过后，才勾选完成并进入下一 phase。

## 总执行规则

- [ ] 每个 phase 开始前先确认改动范围，不混入无关功能。
- [ ] 每个 phase 完成后运行 `code-review-expert`。
- [ ] 每个 phase 完成后补充或更新单元测试。
- [ ] 每个 phase 完成后跑最小端到端 smoke test。
- [ ] 每个 phase 完成后更新 `项目日志.md`。
- [ ] 每个 phase 完成后由用户验收。
- [ ] 用户验收通过后再打勾，并进入下一 phase。

建议每个 phase 的提交节奏：

```powershell
git status -sb
git add <本阶段相关文件>
git commit -m "<阶段性提交信息>"
git push
```

`outputs/` 运行产物默认不提交，除非明确需要作为 demo artifact。

优先级：P0

- [x] 给 `EvidenceItem` 增加 `source_tier` 和 `source_count`。
- [x] 新增 `src/utils/domain_tiers.py`。
- [ ] 改造 `src/evidence/evidence_store.py`：
  - [x] 使用 `classify_url()` 判断 source tier。
  - [x] 使用 `best_tier()` 计算多个来源的最高层级。
  - [x] 低质量/general 来源默认不直接升级为 `evidence_backed`。
  - [x] 官方/学术来源可作为 `evidence_backed`。
  - [x] 多独立来源共识可进入 `verified`，但要保守。
- [x] 改造 trace/report evidence 输出，展示 `source_tier/source_count`。

### 单元测试

- [x] `domain_tiers` 测试：
  - [x] `nasa.gov -> official`
  - [x] `usgs.gov -> official`
  - [x] `openalex.org -> academic`
  - [x] `arxiv.org -> academic`

### 验收标准

## Phase 3：Provider / Profile / ModuleProfile 模型架构优化

优先级：P1

目标：在 `D:\geo-research-agent` 的 model 层基础上继续优化，把多模型切换设计成清晰、可扩展、低耦合的结构。

### 设计目标

模型配置分三层：

```text
Provider: API 服务商、base_url、api_key、adapter
Profile: model、temperature、top_p、max_tokens、context limits
ModuleProfile: planner/solver/summarizer/compressor/reviewer -> profile
```

Policy 只负责一次 LLM 调用，尽量无状态。AgentPool 不复用带运行态污染的 policy。

### 改动范围

- [ ] 保留 `LLMModelFactory` 思路。
- [ ] 抽象或重命名 `VLLMPolicy` 为 `OpenAICompatiblePolicy`。
- [ ] 定义最小 `PolicyAdapter` 接口。
- [ ] provider config 只读取服务商与密钥。
- [ ] profile config 只描述模型和采样参数。
- [ ] module_profiles 只描述模块路由。
- [ ] cache key 排除 `api_key`。
- [ ] 保持旧 `backend/backend_mapping/backend_sampling` 兼容。
- [ ] README 或配置说明中补充三层模型配置图。

### 单元测试

- [ ] 新 YAML 配置可解析。
- [ ] 旧 backend 配置仍可解析。
- [ ] planner/solver/summarizer 能路由到不同 profile。
- [ ] 缺少 API key 时错误信息明确。
- [ ] cache key 不包含 API key。

### 验收标准

- [ ] 用户能通过 YAML 切换不同模块所用模型。
- [ ] 面试时能清楚解释 provider/profile/module_profile/policy 的职责。
- [ ] 用户确认通过。

## Phase 4：可配置外部证据预取

优先级：P1

目标：解决 LLM 不主动调用外部搜索工具的问题，但避免无控制地增加延迟、成本和噪声。

### 改动范围

- [ ] 在 YAML 增加：

```yaml
agents:
  researcher:
    external_prefetch:
      enabled: true
      web_search_top_n: 3
      paper_search_top_n: 3
      timeout_seconds: 15
      max_chars_per_item: 500
```

- [ ] `ResearcherAgent` 执行前按配置预取 `web_search/paper_search`。
- [ ] 预取失败不影响主流程。
- [ ] 预取结果写入 trace：
  - [ ] `external_prefetch_start`
  - [ ] `external_prefetch_result`
  - [ ] `external_prefetch_error`
- [ ] 预取内容注入 prompt，但需要控制长度。
- [ ] 预取结果进入 evidence source，而不仅是 prompt 文本。

### 单元测试

- [ ] 开关关闭时不触发预取。
- [ ] 开关开启时调用 fake web/paper tool。
- [ ] 工具超时不导致 task failed。
- [ ] 注入文本包含 URL/title/source_tier。
- [ ] trace 中有预取事件。

### 验收标准

- [ ] 同一 demo query 下，外部证据可见度提升。
- [ ] `evidence_backed` 占比提高，`speculative` 占比下降或至少不增加。
- [ ] 用户能在 trace 中看懂预取了哪些资料。
- [ ] 用户确认通过。

## Phase 6：实时进度与用户调整方向

优先级：P2

目标：让前端用户看到正在搜索/查询什么，并能中途纠偏；纠偏内容进入后续 task prompt。

### 后端架构

```text
Orchestrator / ToolCallingLoop
        |
        v
progress_callback
        |
        v
ProgressBus -> SSE -> Frontend

Frontend -> POST /tasks/{id}/input -> InteractiveBus
        |
        v
context_modifier -> next task prompt
```

### 改动范围

- [ ] Orchestrator 增加 `progress_callback`。
- [ ] ToolCallingLoop 在工具调用前后发布摘要事件：
  - [ ] task_id

- [ ] 增加 `InteractiveBus`。
- [ ] 增加 `context_modifier`。
- [ ] 用户输入写入下一 task 的 prompt，不强行打断当前 tool call。
- [ ] trace 记录用户指令：
  - [ ] `user_instruction_received`

### 单元测试

- [ ] 用户输入能进入下一 task context。

- [ ] 用户确认通过。

## Phase 7：用户/会话隔离的长期记忆

优先级：P2

目标：理清 user/session/run/task 的关系，避免长期记忆互相污染。

### 数据关系

```text
user_id
  -> session_id
      -> run_id
          -> task_id
```

### 改动范围

- [ ] `SharedMemoryStore` 支持 `session_id`。
- [ ] `MemoryEntry` metadata 增加：
  - [ ] user_id
  - [ ] session_id
  - [ ] run_id
  - [ ] task_id
  - [ ] evidence_level
  - [ ] source_tier
- [ ] `query_by_similarity()` 支持 session/user 过滤。
- [ ] `get_context_for_query()` 支持 session/user 过滤。
- [ ] runner 支持传入 session_id。
- [ ] 失败结果精简写入 memory，避免重复犯错。

### SharedMemoryStore 存储边界

SharedMemoryStore 存：

- 成功任务的代表性 claim。
- verified/evidence_backed 结论摘要。
- rejected 结论和约束。
- failure memory。
- 用户偏好和会话指令摘要。

SharedMemoryStore 不存：

- 完整长报告。
- 大段网页原文。
- 未压缩工具结果。
- 未经过 gate 的 speculative 长文本。

### 单元测试

- [ ] session A 写入的记忆不会被 session B 检索到。
- [ ] global memory 可以被所有 session 读取。
- [ ] failure memory 能按 query 检索。
- [ ] metadata 写入完整。

### 验收标准

- [ ] 多 session 运行不会互相污染。
- [ ] 用户确认通过。

## Phase 8：最小 Wiki 知识库

优先级：P2

目标：把高质量研究结果沉淀为可读、可维护的 Markdown 知识库。

### Wiki 存储边界

Wiki 存：

- 领域实体知识，例如 sensor/method/dataset/analysis/project。
- 高质量研究报告 raw。
- 可复用结论。
- 被明确否定的约束知识，例如 Sentinel-2 不能直接反演 LST。

Wiki 不存：

- 每个 task 的完整 trajectory。
- 大段低质量网页摘录。
- provider/API 错误日志。
- 未达 evidence gate 的 speculative 结论。

### 改动范围

- [ ] 迁移最小 `WikiStore`。
- [ ] 支持用户目录：

```text
data/wiki/users/<user_id>/
  index.md
  raws/
  sensors/
  methods/
  analyses/
  comparisons/
  projects/
  notes/
```

- [ ] 支持 `save_raw()`。
- [ ] 支持 `get_index()`。
- [ ] 支持 `get_context(query)`。
- [ ] Orchestrator 在 planning/task context 中注入 wiki context。
- [ ] Wiki ingest 先只生成 draft，不自动 verified。

### Ingest gate

- [ ] report 不是失败 fallback。
- [ ] 至少一个成功子任务。
- [ ] confidence 达到阈值。
- [ ] 至少一个 `evidence_backed` 或 `verified` evidence。
- [ ] mock source 不进入 Wiki。
- [ ] rejected 可以作为 negative knowledge，但必须显式标记。

### 单元测试

- [ ] Wiki 初始化目录正确。
- [ ] raw report 可保存。
- [ ] index 可读取。
- [ ] get_context 能返回相关页面。
- [ ] speculative 报告不会自动 ingest。
- [ ] rejected negative knowledge 标记正确。

### 验收标准

- [ ] 第一次研究后生成 raw report。
- [ ] 第二次相关研究能读取 Wiki context。
- [ ] 用户能人工检查 Wiki Markdown 文件。
- [ ] 用户确认通过。

## 当前推荐启动顺序

严格顺序：

1. Phase 1：证据质量统一升级
2. Phase 2：搜索结果质量排序
3. Phase 3：Provider / Profile / ModuleProfile 模型架构优化
4. Phase 4：可配置外部证据预取
5. Phase 5：Pipe-based PromptBuilder 增量重构
6. Phase 6：实时进度与用户调整方向
7. Phase 7：用户/会话隔离的长期记忆
8. Phase 8：最小 Wiki 知识库
9. Phase 9：Wiki ingest 与 lint 增强

排序理由：

- 先做 evidence/search，因为它们直接决定研究结果是否可信。
- 再做 model 架构，因为它影响后续所有 agent 的成本和质量控制。
- 再做 external prefetch，因为它依赖 evidence/search 的质量判断。
- 再做 prompt pipe，因为它要承载 prefetch、wiki、用户指令等上下文。
- 最后做交互、长期记忆和 Wiki，因为它们依赖前面这些 gate，否则容易把错误内容长期保存。
