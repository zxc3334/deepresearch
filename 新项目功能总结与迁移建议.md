# 新项目功能总结与迁移建议

对比对象：

- 当前项目：`D:\研究生\找实习\geo-research-agent`
- 新项目：`D:\geo-research-agent`

本次分析范围：

- 重点读取新项目的 `项目日志.md`、`产品日志.md`、`src/`、`configs/`、`app/` 中的后端架构线索。
- 前端、上线部署、Docker、Nginx、Next.js 页面暂不作为当前迁移重点。
- 目标不是把新项目整体覆盖回来，而是筛出真正能增强 GIS/遥感 Deep Research 效果的能力，按优先级迁移。

## 一句话结论

新项目加了很多功能，但真正能让研究效果变好的核心不是前端和产品壳，而是这几类后端能力：

1. 证据质量从单一 `EvidenceLevel` 升级为 `EvidenceLevel + SourceTier` 双维度。
2. 搜索结果增加域名权威度评分和排序，让 NASA/USGS/ESA/OpenAlex 这类来源排在 CSDN/知乎前面。
3. `ResearcherAgent` 增加自动外部证据预取，缓解 LLM 不主动调用 `web_search/paper_search` 的问题。
4. PromptBuilder 改成 pipe-based 结构，静态 prompt 在前、动态上下文在后，更适合后续扩展和 prompt cache。
5. 引入 Wiki 知识库，把高质量研究结果沉淀为可复用知识，而不是只写入一条向量记忆。
6. 记忆系统增加 session/user 隔离，避免不同用户或不同实验互相污染。
7. Orchestrator 增加 progress callback 和 context_modifier，为 SSE 进度、用户中途输入、交互式研究预留了干净接口。

当前建议：先迁移 1-4，再评估 Wiki，最后才考虑产品 API / 前端。

## 值得迁移的好内容

### 1. SourceTier 双维度证据系统

新项目把“这个结论被验证了吗”和“来源质量高不高”拆开了：

- `EvidenceLevel`：结论验证状态。
  - `verified`
  - `evidence_backed`
  - `speculative`
  - `rejected`
- `SourceTier`：来源质量。
  - `official`
  - `academic`
  - `authoritative`
  - `general`
  - `unverified`

这个设计比当前项目更清晰。当前项目里有 URL 就容易被当作 source-backed，但新项目会进一步判断这个 URL 是 NASA 还是 CSDN。这样可以解决一个关键问题：不是所有联网搜索结果都应该提供同等证据权重。

建议迁移内容：

- 从新项目迁移 `SourceTier` 到 `src/orchestrator/schemas.py`。
- 给 `EvidenceItem` 增加：
  - `source_tier`
  - `source_count`
- 修改 `EvidenceItem.to_dict()`，让 trace/report 能展示来源层级。
- 修改 `EvidenceStore.classify()`，让低质量来源默认降级为 `speculative`。

推荐优先级：P0。

### 2. 统一域名分级表 `domain_tiers.py`

新项目新增了 `src/utils/domain_tiers.py`，把域名可信度集中管理：

- `classify_url(url) -> SourceTier`
- `authority_score(url) -> float`
- `best_tier(urls) -> SourceTier`
- `extract_hostname(url) -> str`

这个模块很值得拿过来。原因是搜索排序和证据分级都需要判断域名质量，如果两个模块各维护一套表，后期一定会不一致。

迁移后，整个项目可以形成统一规则：

```text
domain_tiers.py
      |
      +--> WebSearchTool: 搜索结果排序
      |
      +--> EvidenceStore: 证据来源分级
      |
      +--> Trace/Report: 展示 source_tier
```

建议迁移内容：

- 新增 `src/utils/domain_tiers.py`。
- `web_search.py` 使用 `authority_score()` 和 `classify_url()`。
- `evidence_store.py` 使用 `classify_url()` 和 `best_tier()`。

推荐优先级：P0。

### 3. 搜索结果质量评分与排序

新项目在 `BaseWebSearchTool` 中增加了 `_rank_results()`：

```text
final_score = domain_authority_score * 0.6 + keyword_relevance * 0.4
```

并给每条结果补充：

- `_quality_score`
- `_source_tier`

这能直接改善当前项目“联网了，但结果质量不稳定”的问题。比如同样搜索 Landsat thermal band，NASA/USGS/OpenAlex 应该比 CSDN/论坛更靠前。

建议迁移内容：

- `BaseWebSearchTool.MAX_SNIPPET_CHARS = 500`
- `_rank_results(results, query)`
- 所有后端返回后统一调用 `_rank_results()`
- `OfficialSourceSearchTool` 合并去重后也调用 `_rank_results()`
- `EvidenceStore.extract_sources()` 透传 `_quality_score` 和 `_source_tier`

推荐优先级：P0。

### 4. 自动外部证据预取

新项目在 `ResearcherAgent.run()` 前增加了 `_inject_external_evidence()`：

- 自动调用 `web_search(top_n=3)`
- 自动调用 `paper_search(max_results=3)`
- 将结果注入 task prompt 的 `PRE-FETCHED EXTERNAL EVIDENCE` section

这个设计解决了当前项目一个现实问题：即使 prompt 写了“优先使用搜索工具”，LLM 也可能只调用本地 registry 或直接总结。自动预取可以保证每个研究任务至少看到真实外部证据。

但这个功能不能无脑迁移，必须加配置开关。原因：

- 会增加延迟。
- 会增加搜索 API 成本。
- 可能给每个子任务都塞入不相关搜索结果。
- 如果不进入 trajectory/evidence 体系，只注入 prompt，trace 可解释性会变弱。

建议迁移方式：

```yaml
agents:
  researcher:
    external_prefetch:
      enabled: true
      web_search_top_n: 3
      paper_search_top_n: 3
      timeout_seconds: 15
      max_chars_per_item: 500
```

建议同时做两点增强：

- 将预取结果写入 trace，事件名例如 `external_prefetch_result`。
- 将预取证据也转成结构化 evidence source，而不是只放到 prompt 文本中。

推荐优先级：P1。

### 5. Pipe-based PromptBuilder

新项目把 `prompt_builder.py` 改成 pipe 结构：

```text
PromptBuilder
  .pipe("identity", identity_section())
  .pipe("rules", system_rules())
  .pipe("wiki_index", wiki_index())
  .pipe("tool_guide", tool_guide())
  .pipe("task_context", task_context())
  .pipe("user_instructions", user_instructions())
  .build(ctx)
```

这个设计的价值：

- 静态 section 在前，动态 section 在后，适合 prompt cache。
- 每个 section 可以独立启用/跳过。
- 工具说明可以从实际注册工具动态生成，避免 prompt 里写了不可用工具。
- 后续加 Wiki、用户中途输入、领域提示，不需要继续堆大字符串。

但新项目里的实现目前偏简单，直接整体替换可能丢掉当前项目里已经调过的 GIS/RS 规则。建议做“增量迁移”：

- 保留当前 GIS/RS prompt 内容。
- 把组织方式改成 pipe。
- 先迁移 builder 框架，再迁移具体 section。

推荐优先级：P1。

### 6. Wiki 知识库

新项目新增 `src/wiki/`：

- `wiki_store.py`：用户 Wiki 文件系统存储，支持 raw report、页面 CRUD、index、搜索、导出。
- `ingest.py`：把研究报告转成 Wiki 页面。
- `lint.py`：断链、重复、过期、缺元数据、空页检查。

这比单纯向量记忆更适合简历展示，因为它能表达“研究结果沉淀为结构化知识库”。

不过 Wiki 是复杂功能，不建议现在直接全量迁移。原因：

- 它引入新的知识生命周期：draft / confirmed / verified。
- 需要明确哪些 evidence 可以进入 Wiki，避免把幻觉长期保存。
- 需要处理用户隔离、页面命名、重复合并、过期清理。
- 它和 FastAPI 用户系统绑定较多，直接复制会带来产品层依赖。

建议拆成两步：

第一步只迁移最小 WikiStore：

- `save_raw(report_content, query)`
- `get_index()`
- `get_context(query)`
- `list_pages()`
- `read_page()`

第二步再迁移 ingest/lint：

- evidence gate
- page lifecycle
- broken link check
- stale page cleanup

推荐优先级：P2。

### 7. 记忆系统 session/user 隔离

新项目让 `SharedMemoryStore` 支持 `session_id`：

- 初始化时只加载当前 session 的记忆。
- `put()` 时写入当前 session。
- `query_by_similarity()` 支持 `session_filter`。
- `get_context_for_query()` 支持按用户过滤。

这个能力很实用，尤其是后续如果做 Web API 或多用户 demo。即使当前还是 CLI，也可以用不同 session 隔离不同实验。

建议迁移内容：

- `SharedMemoryStore.__init__(session_id="")`
- `query_by_similarity(..., session_filter=None)`
- `get_context_for_query(..., session_filter=None)`
- runner 初始化时传入 session_id

推荐优先级：P2。

### 8. Progress callback 与 context_modifier

新项目在 Orchestrator 中加入：

- `progress_callback`：给 SSE/前端推送任务进度。
- `context_modifier`：在每层任务之间注入用户输入或动态上下文。

这两个设计很有价值，因为它们不是前端逻辑，而是后端 orchestrator 的扩展点。

可以先不迁移 FastAPI/SSE，但保留这两个 hook：

```text
Orchestrator
  -> progress_callback(event)
  -> context_modifier(ctx, subtask)
```

未来要做交互式 Deep Research 时，这两个 hook 比在 orchestrator 里硬编码 API 逻辑更干净。

推荐优先级：P2。

这些对产品化有价值，但当前主要问题是“架构混乱、功能效果没体现出来”。如果现在迁移产品层，会进一步扩大改动面，让核心研究链路更难调试。

建议等核心研究效果稳定后，再单独做产品化分支。

### 2. 前端和 Docker 部署

暂缓。当前目标不是上线，而是把项目讲清楚、跑稳定、体现 GIS/RS deep research 价值。

### 3. 自动 Wiki ingest 全量逻辑

暂缓全量迁移。可以先拿 WikiStore 的读写结构，ingest 先人工或半自动，避免错误知识进入长期知识库。

### 4. 删除工具

新项目删除了 `arxiv_reader`、`browser`、`code_sandbox` 的 Agent 暴露路径，但当前项目可以先不要删。原因：

- `paper_search` 目前还是复用 `ArxivReaderTool`。
- `browser`/`code_sandbox` 以后可能用于真实网页深读和公式验证。
- 清理工具应该等工具调用统计稳定后再做。

## 新项目暴露的问题

新项目虽然功能多，但也有明显风险：

1. 功能堆叠过快，产品层、Wiki、前端、SSE、记忆、证据系统同时推进，导致主研究链路变复杂。
2. 自动预取证据虽然有效，但容易增加 token、延迟和噪声，需要预算控制和 trace 记录。
3. Wiki 自动 ingest 如果门槛不够严，会把错误结论长期保存。
4. PromptBuilder 被重写后，可能丢失当前项目中已经调好的 GIS/RS 细节规则。
5. 测试覆盖仍然不足。新项目有 `tests/unit/test_compression.py`，但核心 evidence/search/wiki/orchestrator 缺少系统化单测。
6. 产品层 API 与研究核心耦合还不够清晰，后续需要把 `src/` 作为稳定内核，`app/` 作为外壳。

## 推荐迁移顺序

### Phase A：先增强证据质量

目标：让联网搜索和论文搜索的价值真正体现在证据分级中。

- [ ] 迁移 `SourceTier`
- [ ] 迁移 `domain_tiers.py`
- [ ] 迁移搜索结果 `_quality_score/_source_tier`
- [ ] 改造 `EvidenceStore` 的 source-tier-aware classify
- [ ] 更新 trace/report evidence 展示
- [ ] 增加单测：NASA、OpenAlex、Wikipedia、CSDN 的分级结果

验收标准：

- CSDN/知乎类来源不能直接让结论变成 `evidence_backed`。
- NASA/USGS/ESA/OpenAlex 能显示清晰的 source tier。
- Trace report 里能看到每条 evidence 的来源层级。

### Phase B：再增强 Agent 获取证据能力

目标：避免 LLM 不主动调用外部搜索工具。

- [ ] 增加 `external_prefetch` 配置
- [ ] 在 `ResearcherAgent` 中按配置预取 `web_search/paper_search`
- [ ] 预取结果写入 trace
- [ ] 预取结果进入 evidence source，而不只是 prompt 文本
- [ ] 设置字符预算，避免 prompt 膨胀

验收标准：

- 同一个 demo query 下，`evidence_backed` 数量提升，`speculative` 占比下降。
- trace 中能定位预取证据来自哪些 URL/paper。
- 预取失败不影响主流程。

### Phase C：整理 Prompt 架构

目标：降低 prompt 维护成本，为 Wiki、用户输入、领域提示预留位置。

- [ ] 引入 pipe-based PromptBuilder 框架
- [ ] 保留当前 GIS/RS prompt 内容
- [ ] 将工具说明改为从 ToolRegistry 动态生成
- [ ] 增加 debug 输出，记录每个 prompt section 的开关和长度

验收标准：

- 生成的 system prompt 结构清晰。
- prompt 中只出现真实注册的工具。
- 静态 section 和动态 section 分离。

### Phase D：最小化 Wiki 知识库

目标：把高质量报告沉淀为可复用知识，但不引入完整产品层。

- [ ] 迁移最小 `WikiStore`
- [ ] 支持 raw report 保存
- [ ] 支持 `get_index()` 和 `get_context(query)`
- [ ] Orchestrator planning/task context 注入 wiki context
- [ ] 只允许 `verified/evidence_backed` 内容进入 Wiki

验收标准：

- 第二次相关 query 能读到第一次报告沉淀的知识。
- Wiki 不保存失败报告和低置信 speculative 报告。
- Wiki 文件结构可人工检查。

### Phase E：用户隔离和交互 hook

目标：为产品化保留接口，但不迁移完整产品层。

- [ ] `SharedMemoryStore` 支持 session_id
- [ ] runner 支持传入 session_id
- [ ] Orchestrator 增加 `progress_callback`
- [ ] Orchestrator 增加 `context_modifier`

验收标准：

- 不同 session 的记忆检索互不污染。
- CLI 也能通过 callback 打印进度事件。

## 当前最应该做的下一步

建议下一步先做 Phase A：证据质量统一升级。

理由：

1. 它直接改善研究效果，而不是增加产品外壳。
2. 改动边界相对清晰：`schemas.py`、`domain_tiers.py`、`web_search.py`、`evidence_store.py`、trace/report。
3. 它能让已有 `web_search`、`official_doc_fetcher`、`paper_search` 的价值真正可见。
4. 它是后续自动预取、Wiki ingest 的前提。没有可靠的来源分级，后续知识库会更容易污染。

建议暂时不要先迁移 Wiki 或 FastAPI。Wiki 依赖高质量 evidence gate，FastAPI 依赖稳定内核。先把内核证据质量做好，后面功能才不会越加越乱。

## 2026-06-01 补充：四个仍然值得优化的方向

结合 `D:\geo-research-agent` 新项目现状，除了前面列出的 evidence/search/prompt/wiki 能力外，还有四个方向值得明确纳入后续改造计划。

### 1. 用户实时调整研究方向

目标不是简单做“前端进度条”，而是让用户在 Deep Research 运行中看到当前搜索/查询方向，并能在发现方向偏离时插入修正指令。

新项目中已有相关雏形：

- `progress_callback`：Orchestrator 向外发布阶段、当前 task、已完成子任务数。
- `SSE`：后端通过 server-sent events 推送进度。
- `InteractiveBus`：接收用户中途输入。
- `context_modifier`：在层与层之间把用户输入注入下一个 task context。
- `PromptBuilder.user_instructions` pipe：把用户实时指令写入后续 prompt。

建议在此基础上优化：

- 不只推送 `task_id/status`，还要推送“正在查什么”的摘要：
  - 当前子任务描述。
  - 当前工具名。
  - 当前搜索 query。
  - 搜索结果 top titles / source tiers。
  - 当前 evidence 摘要。
- 用户输入不应立刻打断正在执行的 tool call，而是进入 `pending_user_instruction` 队列。
- 在下一个可控边界注入：
  - 当前 task 结束后。
  - 当前 DAG layer 结束后。
  - replanning 前。
- 用户修正指令必须写入 trace，便于复盘：
  - `user_instruction_received`
  - `user_instruction_applied`
  - `context_modified`

推荐架构：

```text
ToolCallingLoop / Orchestrator
        |
        | progress events
        v
ProgressBus -> SSE -> Frontend
        ^
        | user instruction
InteractiveBus <- POST /tasks/{id}/input
        |
        v
context_modifier -> next task prompt
```

迁移优先级：P2。它对产品体验价值高，但应在 evidence/search 质量稳定后做。

### 2. 不同模型切换架构

新项目的模型层比当前项目更清楚：从旧的 `backend` 命名，升级为三层：

```text
Provider -> Profile -> ModuleProfile -> Policy
```

含义：

- `Provider`：API 服务商和鉴权信息，例如 DeepSeek、OpenAI、MiMo、本地 vLLM。
- `Profile`：某个模型及采样参数，例如 `mimo-reasoner`、`deepseek-fast`。
- `ModuleProfile`：模块到 profile 的映射，例如 planner 用强模型、solver 用便宜模型、summarizer 用长上下文模型。
- `Policy`：最终可调用的一次 LLM 调用对象。

新项目现状已经比当前结构更好，但还可以继续优化：

- 把 `VLLMPolicy` 重命名或抽象为 `OpenAICompatiblePolicy`，避免把所有 OpenAI-compatible 服务都叫 vLLM。
- 引入 `PolicyAdapter` 接口，为未来 Anthropic/Gemini/本地模型留入口。
- Provider 只负责 base_url/api_key/adapter，不混入 temperature/max_tokens。
- Profile 只负责 model/sampling/context limits，不直接读取 env。
- Module routing 只描述 planner/solver/summarizer/compressor/reviewer 等模块用哪个 profile。
- policy cache key 必须排除 `api_key`，但包含 provider/profile/model/sampling。
- 每个 agent pool executor 默认使用独立 policy 实例，避免 mutable state 污染。

推荐配置形态：

```yaml
model:
  providers:
    mimo:
      adapter: openai_compatible
      env_prefix: MIMO
      default_base_url: https://token-plan-sgp.xiaomimimo.com/v1
    deepseek:
      adapter: openai_compatible
      env_prefix: DEEPSEEK

  profiles:
    planner_strong:
      provider: mimo
      model: mimo-v2.5-pro
      temperature: 0.1
      max_tokens: 4096
    solver_balanced:
      provider: deepseek
      model: deepseek-chat
      temperature: 0.2
      max_tokens: 4096
    summarizer_long:
      provider: mimo
      model: mimo-v2.5-pro
      temperature: 0.1
      max_tokens: 8192

  module_profiles:
    planner: planner_strong
    solver: solver_balanced
    summarizer: summarizer_long
    compressor: solver_balanced
```

迁移优先级：P1。它不直接提升证据质量，但能显著降低多模型实验和成本控制复杂度。

### 3. Wiki、长期记忆、用户/会话关系要理清

当前应该把三类状态明确分开：

```text
User
  |
  +-- Session / Run
  |     |
  |     +-- short runtime context
  |     +-- task results
  |     +-- trace events
  |
  +-- SharedMemoryStore
  |     |
  |     +-- evidence-aware memory entries
  |     +-- failures / lessons
  |     +-- retrieval context
  |
  +-- Wiki Knowledge Base
        |
        +-- curated pages
        +-- raw reports
        +-- index
        +-- lifecycle metadata
```

建议定义：

- `user_id`：用户维度，拥有自己的 Wiki 和长期记忆。
- `session_id`：一次研究会话或实验维度，用于隔离临时结果和检索范围。
- `run_id`：一次 orchestrator run，用于 trace 和输出目录。
- `task_id`：DAG 子任务维度。

建议命名：

```text
user_id = "user:xxx"
session_id = "user:xxx/session:yyy"
run_id = "run:timestamp-or-uuid"
```

迁移时不要把 Wiki 和 SharedMemoryStore 混成一个东西。Wiki 是可读、可维护、结构化知识；SharedMemoryStore 是检索型记忆和经验记录。

迁移优先级：P2。

### 4. Wiki 知识库和 SharedMemoryStore 分别存什么

建议明确边界如下。

Wiki 知识库存：

- 高质量、可复用、可人工阅读的领域知识。
- 适合长期沉淀的结论，例如：
  - Landsat 8/9 TIRS 可用于 LST 反演。
  - Sentinel-2 没有热红外波段，不能直接反演 LST。
  - NDVI/NDBI 的公式、适用条件、误用风险。
  - 某个项目的最终研究方案。
- 存储形式：
  - Markdown 页面。
  - index。
  - frontmatter/status。
  - raw reports。
- 生命周期：
  - `draft`
  - `confirmed`
  - `verified`
  - `stale`

SharedMemoryStore 存：

- 每次运行中可用于语义检索的记忆片段。
- 适合给 planner/replanner 提供上下文的经验。
- 包括：
  - 成功任务的 primary claim。
  - evidence-backed/verified 结论摘要。
  - rejected 结论，用于避免重复错误。
  - failure memory，例如 API timeout、tool failure、provider error。
  - 用户偏好或本次 session 指令摘要。
- 存储位置：
  - 当前项目已有 SQLite memory DB。
  - 每条记录携带 `session_id/user_id/task_id/evidence_level/source_tier` metadata。

Wiki ingest 应该由 evidence gate 控制：

```text
只有满足以下条件才允许自动进入 Wiki：
1. report 不是 fallback-only 失败报告
2. 至少有一个成功子任务
3. confidence 达到阈值
4. 至少包含 evidence_backed 或 verified evidence
5. 来源不是纯 mock
6. rejected 可作为 negative knowledge 保存，但必须标记为 rejected/constraint
```

建议默认策略：

- 自动保存 raw report。
- 自动 ingest 只生成 draft。
- confirmed/verified 必须依赖重复出现或人工确认。
- speculative 默认不进入 Wiki，除非用户显式保存为 note。

迁移优先级：P2。
